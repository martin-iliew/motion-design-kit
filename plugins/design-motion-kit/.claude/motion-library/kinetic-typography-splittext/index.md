# Kinetic Typography with SplitText

**Framework:** GSAP
**Category:** typography
**2026 Relevance:** Kinetic typography is one of the defining web design directions of 2026, with text motion replacing static headings as the primary visual hierarchy tool on landing pages and editorial sites. GSAP's SplitText plugin — now free for all following Webflow's acquisition — received a major v3.13 rewrite with built-in masking, `autoSplit`, and aria accessibility, making character/word/line stagger animations both easier to implement and safer to ship.

## Description

Kinetic typography with SplitText works by decomposing a text element into granular DOM nodes — individual characters (`split.chars`), words (`split.words`), or lines (`split.lines`) — which GSAP can then target as an array. Each sub-element becomes an independently animatable unit, enabling staggered entrance choreography that would be impossible to achieve with CSS alone. The v3.13 rewrite cut the plugin's bundle size roughly in half and introduced the `mask` parameter, which wraps each split unit in an overflow-clipping container automatically, eliminating the need for manual `overflow: hidden` parent markup that was previously required for the classic "text slides up from below the baseline" reveal.

The most impactful 2026 patterns combine SplitText with ScrollTrigger for scroll-driven reveals, using `yPercent: 100` on `split.words` inside masked containers to achieve the editorial "curtain lift" effect seen across agency and portfolio sites. Character-level animations add a second layer of expressiveness: `rotationX: -90` on `split.chars` creates a perspective-flipping entrance that communicates brand confidence without requiring 3D CSS transforms on the parent. The `stagger` object's `from: "center"` and `from: "random"` options produce distinct emotional reads — centered staggers feel deliberate and cinematic while random staggers feel energetic and playful.

Performance and accessibility are non-negotiable constraints in 2026. SplitText v3.13+ automatically adds `aria-label` to the original element and `aria-hidden` to all generated child spans, so screen readers consume the original text without being flooded by hundreds of individual character nodes. The `autoSplit: true` option uses a `ResizeObserver` and `document.fonts` loading event to re-split text on layout reflow and font load, preventing the broken wrapping that occurs when custom fonts load after the initial split. All animation code must be wrapped in `gsap.matchMedia()` to respect `prefers-reduced-motion`, and SplitText instances must be explicitly reverted on component unmount to avoid memory leaks and orphaned DOM nodes.

## Do's

- Always call `gsap.registerPlugin(SplitText, ScrollTrigger)` once at the app entry point, not inside components, to avoid duplicate plugin registrations.
- Use `mask: "lines"` or `mask: "words"` on the SplitText constructor instead of manually wrapping elements in `overflow: hidden` containers — the built-in mask parameter is more reliable with nested markup and variable fonts.
- Use `autoSplit: true` for any text that may reflow due to responsive breakpoints or late-loading web fonts, so split indices stay correct after resize.
- Set `stagger: { each: 0.02–0.05 }` for character-level animations and `stagger: { each: 0.06–0.12 }` for word/line animations — tighter staggers feel energetic, wider staggers feel cinematic; calibrate to the emotional register of the brand.
- Always call `split.revert()` in your cleanup function (returned from `gsap.matchMedia()` or `ctx.revert()` in React) to restore the original DOM and prevent orphaned span elements accumulating on route changes.

## Don'ts

- Do not call `new SplitText()` before fonts have loaded — custom web fonts change character widths and will produce incorrect line breaks; use `document.fonts.ready.then(...)` or rely on `autoSplit: true`.
- Do not apply CSS `transition` properties to the same elements GSAP is animating — competing animation owners cause unpredictable jank and fight for the transform matrix.
- Do not skip the `prefers-reduced-motion` guard; character-by-character flicker can trigger vestibular disorders, and omitting the guard is an accessibility failure, not merely a best practice gap.
- Do not stagger more than ~150–200 characters at once with `rotationX` or `rotationY` — perspective-based transforms on hundreds of nodes are GPU-expensive and will drop frames on mid-range mobile devices; prefer word or line splits for body text.
- Do not use `gsap.from()` on elements that start with `opacity: 0` set via CSS — the animation completes at `opacity: 1` correctly but the element can flash invisible momentarily; use `gsap.set()` to establish the starting state before the tween, or use `gsap.fromTo()` with explicit start and end values.

## Best Practices

**Accessibility.** SplitText v3.13+ handles ARIA automatically: it writes `aria-label` containing the full original text onto the parent element and stamps `aria-hidden="true"` on every generated character/word/line span. This means screen readers announce the complete sentence rather than individual letters. Your responsibility is the motion layer: wrap all tween code inside `gsap.matchMedia()` with a `(prefers-reduced-motion: no-preference)` condition, and provide an instant `gsap.set()` fallback in the `(prefers-reduced-motion: reduce)` branch that ensures the text is visible and correctly positioned without any animation. Never rely on the browser's CSS `@media (prefers-reduced-motion)` alone when using JavaScript-driven animations — the GSAP `matchMedia` API is the correct intercept point because it coordinates cleanup of ScrollTrigger instances as well.

**Performance.** Character-level animations create many DOM nodes — a 60-character headline produces 60 individual `<div>` elements. Limit per-element property animation to `y`, `yPercent`, `opacity`, `rotation`, `rotationX`, `rotationY`, `scale`, and `transformOrigin`, all of which are GPU-composited via the CSSOM transform matrix. Avoid animating `font-size`, `letter-spacing`, `width`, or `height` on split nodes as these trigger layout reflow on every frame. For sections with multiple kinetic blocks, use `ScrollTrigger.batch()` rather than creating individual ScrollTrigger instances — batch coalesces IntersectionObserver callbacks and significantly reduces per-frame overhead on pages with dense typography. Call `split.revert()` as soon as the entrance animation completes if the element will not be re-animated, to remove the extra spans from the DOM and reduce node count.

**React and SPA cleanup with `gsap.context()`.** In React, always initialize SplitText and its accompanying tweens inside a `gsap.context(callback, scopeRef)` call within `useLayoutEffect`. The context scope-pins all GSAP selectors to the component's DOM subtree, preventing selector collisions with identically named elements elsewhere on the page. Returning `ctx.revert()` from the `useLayoutEffect` cleanup function kills all tweens, ScrollTrigger instances, and `matchMedia` listeners created inside that context in a single call — including the SplitText revert — making it the safest single-line teardown pattern for component-based architectures. In Next.js App Router, prefer `useIsomorphicLayoutEffect` (a small utility that switches between `useLayoutEffect` and `useEffect` based on `typeof window`) to avoid SSR hydration warnings, since `useLayoutEffect` fires synchronously and GSAP's DOM operations are client-only.
